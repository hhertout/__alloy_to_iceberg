# Motivation

We operate a Grafana stack in production, in what we can consider as a large cluster (60 nodes with 32 CPU and 256GB RAM).

We already have good monitoring and incident detection in place. However, in some cases related to the volume of data we ingest or user query patterns, we can experience performance issues that require immediate attention to avoid user impact.

A major incident occurred in production despite our existing monitoring and controls. My involvement in troubleshooting this incident provided valuable insights into the stack's behavior. I was able to correlate specific metric combinations with the underlying issue and understand the root cause. This experience has enabled me to respond faster to similar incidents.

Following that incident, I conceived the idea of building a deep learning model to enhance our stack troubleshooting and monitoring capabilities, and to predict potential issues before they occur. The goal was to build a model capable of understanding the entire system based on a comprehensive metric set that would be difficult for humans to interpret quickly. Since we already have the necessary data, metrics, and logs available through our metamonitoring infrastructure, leveraging this data to train a predictive model made sense.

The objective was not simply to detect seasonality or identify anomalies in individual metrics, but to build a model that understands the system holistically and can predict potential issues before they happen.

I presented this idea to my team, and they were interested and supportive. Though I had no prior experience in machine learning or data science, I was motivated to explore this field and develop the necessary skills. I decided to proceed with the project, learning as I went along, and building the model step by step.

# Building the model

## Step 1 - Building a pipeline to retrieve data, clean it and send it to longterm storage

The first step I decided was to build as fast as possible, a pipeline to retrieve data from our Grafana stack, to clean it and send it to longterm storage.

### Tech stack

- Grafana / Prometheus / Loki
- Python
- Polars for data manipulation
- Azure Blob Storage for longterm storage (equivalent to AWS S3)

### Key metrics definition

The choice of the metrics to train the model is essential. So I decided to choose many metrics related the monitoring we have in place, and moreover based on my personal experience with Grafana.

Though, I decided to choose the following metrics, which I considered relevant for the model training:

- Alloy
    - Queue length - represents the number of pending queries in the queue, which can indicate performance issues or bottlenecks.
    - Error logs - representative of the error happening at the highest level of the stack. The exact error show the issue and can be interpreted/converted to train the model. The idea behind is to be able to link a specific error to a specific combination of metrics.

- Mimir
    - Ingestion - measures the rate at which data is being ingested into Mimir, which can help identify potential issues with data flow or performance. These metrics are based on the distributor and ingester components of Mimir, which are responsible for receiving and processing incoming data.
    - Read - provide to the user the data they want to see, and the performance of this component is crucial for the user experience. These metrics are based mainly on the querier, and store-gateways component of Mimir, which is responsible for processing user queries and returning results.

- Loki
    - Write - measures the throughput and latency of log ingestion, which indicates how quickly logs are being written to Loki. These metrics track the distributor and ingester components, revealing bottlenecks in the log pipeline that could impact log availability and searchability.
    - Read - reflects the performance of log queries and the ability of Loki to serve log data to users. These metrics focus on the querier and store-gateways components, which directly affect the user experience when searching or viewing logs.
    - Compaction - this process is crucial for maintaining Loki's performance and storage efficiency. Metrics related to compaction can indicate how well Loki is managing its log data, and whether there are any issues that could lead to increased latency or storage problems.

- Tempo
    - Write - measures the ingestion rate and latency of distributed traces, which is critical for understanding the system's ability to handle incoming trace data. These metrics monitor the distributor and ingester components, helping identify capacity issues before they impact trace collection.
    - Read - reflects the performance of trace queries and retrieval, ensuring that traces can be accessed quickly for troubleshooting and analysis. These metrics are based on the querier and store-gateways components, which determine how efficiently trace data can be queried and displayed to users.

I volontarily choose a lot of metrics, even if I don't use them all for the training, because I wanted to have a good dataset to train the model, and to be able to test different combinations of metrics to see which one is the most relevant for the model training without the need to wait to have a good amount of data again.

### Flow

The flow is really simple.

```
   queries.yaml        Grafana (Prometheus + Loki)              Azure Blob Storage
 ┌──────────────┐           ┌──────────────────────┐          ┌─────────────────────────────────────┐
 │  PromQL &    │──parse──▶ │  /api/ds/query       │──upload─▶│  container/chunks/                  │
 │  LogQL       │           │  (previous day)      │          │   chunk_dataframe_YYYYMMDD.parquet  │
 │              │           └──────────────────────┘          └─────────────────────────────────────┘
 └──────────────┘                     │
                                      ▼
                              Polars DataFrames
                              (merge, resample,
                               process, parquet)
```
